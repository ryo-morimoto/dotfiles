# 開発ワークフロー設計（仕事向け）

Status: draft（運用しながら更新する。強制フォーマットにしない）

## 前提

- 開発の意思決定は 1 人で行う。
- アーキテクチャに関する変更は 2 人でレビューする。
- 利用者のオペレーションに関する変更は、要件段階で 2 人で妥当性をレビューする（リリース時では遅い）。
- 範囲: 要件〜実装＋リリース・運用。
- フィードバックの主眼: フェーズ間の手戻り制御、成果物への HitL レビュー、プロセス自体の改善。

## 全体像

```
[受付] → [オペレーション設計] → [基本設計] → [詳細設計] → [タスク化・実装] → [リリース] → [運用]
   ↑           ↑                    ↑              │                                        │
   │           │                    └── 設計不備 ←──┘                                        │
   │           └── オペレーションの穴 ←──────────────────────────────────────────────────────┘
   └── 運用からの気づき・利用者 FB は新チケットとして受付へ
                        ↓
                  [振り返り] → プロセス自体を更新（codify）
```

## 変更分類（受付時に仮判定、設計・実装時に再確認）

| 変更種別 | ゲート位置 | レビュー対象物 | 見ること |
|---|---|---|---|
| 通常の開発判断 | 実装（CI + セルフレビュー） | PR | — |
| アーキテクチャ変更 | 基本設計の出口・2 人 | 設計文書 / ADR | 技術判断の妥当性 |
| 利用者オペレーション変更 | オペレーション設計の出口・2 人 | オペレーション定義（操作フロー含む） | オペレーションとして成立するか |

PR 時に分類に気づくのが一番手戻りが大きいため、受付時に判定する。
途中でアーキ変更に化けた場合は、その時点で該当ゲートに戻す。

## フェーズ定義

### 0. 受付

- 成果物: チケット（目的・スコープ外・制約・検証基準・変更分類）。
- 出口基準: WHEN/THEN の検証基準が書けている。書けない場合は探索不足なので調査タスク化する。

### 1. オペレーション設計

- 対象業務をオペレーション単位で定義し、ワーキングユニットに分割する。
- 各ワーキングユニットは自己完結させる:
  - ユニット外のコンテキストは**前提条件**と**完了条件**として表現する。
  - 完了条件は後続処理との接続で検証できる形にする（例: 後続機能の待ちリストに表示されること）。
  - 依存が切れていれば、他のオペレーションの状態を気にせずユニット単体でタスクを完了できる。
- 出口基準: 依存が切れている・抜け漏れがない。利用者オペレーション変更に該当する場合は 2 人レビュー承認。
- 戻り先: 受付（目的から再定義）。

### 2. 基本設計

- 成果物: 状態遷移（トリガー・条件・例外）と、状態ごとのデータ設計。
- データ構造は 2 つの側面で分けて整理する:
  - **オペレーションレベル**（グローバル変数に相当）: オペレーション全体で必要なデータ一覧。
  - **ワーキングユニットレベル**（ローカル変数に相当): このユニットで定義・修正する部分。
  - ユニット単位で決めて合成する（ボトムアップ）。抜け漏れチェックは両側面で行う。
- ドキュメント構造はユニット非依存のセクション名で統一する（例:「条件」「例外」）。
  構造が統一されると認知負荷が下がり、レビュー可能になる。
- **この段階で実現可能性を担保する**。後段の詳細設計でロジックが最後になっても
  問題ないのは、ここで検証済みだから。
- 出口基準: アーキテクチャ変更に該当する場合は 2 人レビュー承認。
- 戻り先: オペレーション設計。

### 3. 詳細設計（3 段階、粒度は大 → 小）

1. **ページツリー**（オペレーションレベル）: どのページでこのオペレーションを実現するか。
   → ここでファーストレビュー。ページの区切りという大きな認識がずれていると、
   その下の詳細レビューは意味がないため。
2. **ページ詳細**（ワーキングユニット単位）: 何のために何を表示するか。UI 目線で必要な
   パラメーター。画面レイアウトまでは作らない（粒度が細かすぎると変更に弱くなる）。
3. **ファイル・ディレクトリ構造**: ページを実現するためにどこに何を置くか。
   不明点は AI に聞いて把握する。疑問に思う＝自分の考え方と違う箇所なので、
   妥当なら合わせ、自分の方が良ければ変える。

- 位置づけ: 詳細設計は**方向性が合っているかを確認するための基準**。
  実装中に変えるのは良いこと。無いのは博打（合っているかのチェックができず再現性もない）。
- 精度は 8 割で前へ進む。ズレが大きそうな箇所（何のために何を表示するか等)は
  自分の想定を書いて早めにレビューに出す。

### 4. タスク化・実装

- ページ＋ファイル構造が決まればタスク化できる。できない場合は詳細設計に穴がある。
- タスクの検証方法に**後続処理との接続チェック**を含める。
- 実装中の詳細設計からの逸脱は OK（設計文書は方向性の基準であって契約ではない）。
  承認済みオペレーションからの逸脱はオペレーション設計に戻して再レビュー。

### 5. リリース

- 承認済みオペレーションとの一致をセルフチェックし、リリースノートを作成する。
- 新たなレビューは設けない（オペレーションの妥当性は要件段階で担保済み）。

### 6. 運用

- 監視・利用者 FB を記録する。気づきは新チケットとして受付へ還流する。

### 7. 振り返り（プロセス改善）

- 「最初に詰まったこと」と「最終的に効いたこと」をペアで記録する。
- 「8 割を 9 割に上げるツール」を増やす: レビュー視点の追加、チェックリスト、
  ドキュメント構造の改善など。
- 機械化できる学びは CI / lint / hooks へ、判断が必要な学びは本文書の更新へ振り分ける。
- 本文書は自分用の運転手順書として更新し続ける。チームへの強制フォーマットにはしない
  （変更に弱くなる・ドキュメント構造から思考の抜けを読み取れなくなるため）。

## レビュー運用

- 成果物単位（文書・PR）の非同期レビューを基本にする: Slack に貼る → 非同期レビュー、
  タイミングが合えばハドル。
- レビューは粒度の大きい認識合わせから段階的に行う（ページ区切り → 詳細）。
  大きな認識のズレを先に潰すことが手戻り制御の実体。

## スタンス: 8 割原則

- 8 割の確度で進め、残り 2 割のズレは許容する。0 で進むのは博打、100 を待つのは遅い。
- 「自分の 100% の基準を上げる」（レビューを通して学ぶ）と「現状の 100% を達成する」は
  分けて考える。前者はレビューで、後者は自分のワークフローで担保する。
