Lesson 0001

Kodyは何の土台か

このレッスンのゴールは、Kodyを「Hermesの代替」と雑に見るのをやめて、 どの層をKodyに任せ、どの層を外部agentに任せるかを判断できるようにすることです。

今日の勝ち筋

まず観察から始めます。Kodyの一次資料は、Kodyを「Cloudflare WorkersとMCP上のpersonal assistant platform」と説明し、public MCP surfaceを小さく保って、`search` と Code Mode `execute` の背後にcapability graphを置く設計だと説明しています。 参照: Kody README, project intent。

ここからの推論はこうです。公開されているKodyの中心設計は、AI hostが使う「道具、記憶、鍵、保存済みコード、job、UI」を MCP経由で提供することです。一方、repoをcloneして長時間推論し、実装し、PRを作るcoding agent loopは、Kodyの中心的なprimitiveとしてはまだ現れていません。このため現時点では、Kodyを agent実行基盤そのもの ではなく、 agent実行基盤が使うMCP-nativeな自動化基盤 と見るのが安全です。

3層で見る

AI host ChatGPT、Claude、Cursorなど。ユーザーと会話し、MCP clientを通じてKodyを呼ぶ。
→
Kody `search`、`execute`、packages、secrets、memory、jobs、repo sessionsを提供する。
→
External systems GitHub、Cursor Cloud Agents、Cloudflare、メール、任意のAPI。

MCP公式資料は、MCPをAI applicationsと外部systemsをつなぐ標準と説明しています。 Kodyのproject intentは、この標準の上で、hostごとに巨大なtool catalogを持たず、 少数の入口から多くのcapabilityへ到達させる方針だと読めます。 参照: MCP introduction, MCP tools spec。

Hermesとの作業仮説

注意: この比較は、まだHermes側の一次資料で検証していない作業仮説です。 ここで確かに言えるのは、Kody側の一次資料と実装から見える役割だけです。

Hermesのtooling/control plane

Kodyが近い可能性がある層。Kodyにはsecret、外部API呼び出し、記憶、保存済みworkflow、job、UI、repo補助がある。

Hermesのagent execution

Kodyの主要primitiveとしては前面に出ていない層。planner、executor、reviewer、長期agent loop、coding runtimeは別基盤に任せる候補。

実用上の見方

現時点の運用仮説では、Kodyを「作業者」より「作業者が使う作業場」と見る。作業者候補はCursor/Codexなど。

`start_coding_agent` はどこに置くか

最初の実験先としては saved package が有力です。Kody docsは「agent turnsはbuilt-in runtime primitiveではなくpackage-owned behavior」と説明しています。そこから、Cursor Cloud Agentを起動する workflowも、まずpackageとして保存し、`execute`やpackage invocationから呼ぶ設計を試す価値があります。 参照: execute and workflows。

GitHubやCursor Cloud AgentsへのPOST/PATCH/DELETE、またはquota-consuming agent operationは、 実行前にmethod、path、bodyを確認するのがKodyの安全方針です。 参照: mutating actions。
