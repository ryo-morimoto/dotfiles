# agent-infra/sandbox

## Summary

AIO Sandbox is an "all-in-one" agent workspace: one Docker image
(`ghcr.io/agent-infra/sandbox`) runs Browser+VNC, Shell, Filesystem, Jupyter,
Node.js, VSCode-server, a port preview proxy, and an MCP hub behind a single
HTTP port (8080). The container *is* the trust boundary; the agent talks to
it remotely via a generated REST SDK (Python / TS / Go) — there is no
fine-grained per-call permission model, only "what's reachable inside the
box."

## Architecture

Single container, multi-process. From `docker-compose.yaml`:

- Public port: `8080` (PUBLIC_PORT). Everything is multiplexed behind it.
- Internal services and ports:
  - `MCP_HUB_PORT=8079` — MCP aggregator
  - `SANDBOX_SRV_PORT=8091` — main REST API (`/v1/*`)
  - `AUTH_BACKEND_PORT=8081`, `WEBSOCKET_PROXY_PORT=6080`
  - `VNC_SERVER_PORT=5900`, `BROWSER_REMOTE_DEBUGGING_PORT=9222`
  - `JUPYTER_LAB_PORT=8888`, `CODE_SERVER_PORT=8200`, `GEM_SERVER_PORT=8088`
  - Per-MCP-server side-cars: `MCP_SERVER_BROWSER_PORT=8100`,
    `MCP_SERVER_MARKITDOWN_PORT=8101`, `MCP_SERVER_CHROME_DEVTOOLS_PORT=8102`
  - `TINYPROXY_PORT=8118` — outbound HTTP proxy (used to route container
    traffic through `PROXY_SERVER` env)
- `WAIT_PORTS=8079,8091` — container's "ready" signal is MCP hub + REST API.
- Boots with `--security-opt seccomp=unconfined`, `shm_size=2gb`,
  `mem_limit=8g`, `cpus=4`. Workspace lives at `/home/gem` (`WORKSPACE`).

The architecture diagram in the README:

```
Browser+VNC | VSCode | Shell | File Ops | MCP Hub + Sandbox Fusion |
Preview Proxy + Service Monitoring
```

The Dockerfile is *not* in the public repo (the `docker/` directory is
empty in this clone — image is built privately and published to GHCR).

## Tool Surface

Two layers, both HTTP, both fronted by port 8080:

1. **REST API** under `/v1/*`, OpenAPI-generated by Fern. The Python SDK
   (`sdk/python/agent_sandbox/client.py:32`) exposes resource clients:
   `sandbox`, `shell`, `bash`, `file`, `jupyter`, `nodejs`, `code`,
   `browser`, `browser_page`, `browser_tabs`, `browser_cookies`,
   `browser_state`, `browser_network`, `browser_captcha`, `mcp`, `proxy`,
   `skills`, `util`, `auth`. Sample method shapes:
   - `shell.exec_command(command, id?, exec_dir?, async_mode?, timeout?,
     hard_timeout?, strict?, ...)` — supports SSE streaming via
     `Accept: text/event-stream`. Also `create_session`, `view`,
     `wait_for_process`, `write_to_process`, `kill_process`,
     `get_terminal_url`.
   - `file.{read_file, write_file, replace_in_file, search_in_file,
     find_files, grep_files, glob_files, upload_file, download_file,
     list_path, str_replace_editor}` — note `str_replace_editor` mirrors
     Anthropic's text-editor tool.
   - `sandbox.{get_context, get_python_packages, get_nodejs_packages}`
     for env introspection (returns `home_dir`, etc.).
   - `proxy.{add_mapping, remove_mapping, list_mappings, add_exclude,
     remove_exclude, diagnose}` — manages the in-container preview proxy.
   - `skills.{register_skills, list_metadata, clear_skills, delete_skill,
     get_content}` — Claude Skills-style bundle hosting.

2. **MCP Hub** at `/mcp` (port 8079 internally). A single REST aggregator
   over multiple side-car MCP servers (browser, file, shell, markitdown,
   chrome-devtools). Schema (`sdk/python/agent_sandbox/mcp/client.py`):
   - `mcp.list_mcp_servers()` → list of registered server names.
   - `mcp.list_mcp_tools(server_name)` → tool schemas.
   - `mcp.execute_mcp_tool(server_name, tool_name, request: dict)` →
     `CallToolResult`.
   So MCP is exposed as an HTTP RPC, not a stdio MCP transport. Agents
   either call `/mcp` over HTTP, or talk MCP-over-SSE/streamable directly
   to the hub if their client supports it (the README markets `/mcp` as
   an MCP endpoint).

## Policy / Config

There is essentially no per-call policy engine. Configuration is at the
container boundary:

- `seccomp=unconfined` is the documented default, i.e. the project trusts
  Docker's process isolation but explicitly drops kernel syscall filtering.
- `WORKSPACE` env scopes the agent's home dir but file APIs accept absolute
  paths and have no stated allow-listing.
- Egress: `PROXY_SERVER` env routes outbound traffic through a configurable
  HTTP proxy (`TINYPROXY_PORT=8118`); `DNS_OVER_HTTPS_TEMPLATES` enforces
  DoH. There's no per-domain allow/deny visible in the SDK.
- `JWT_PUBLIC_KEY` enables JWT validation on incoming requests;
  `auth.create_ticket` / `auth.authenticate` is the entry point. This is
  *transport*-level authn between agent and box, not behavioural policy.
- `DISABLE_JUPYTER`, `DISABLE_CODE_SERVER`, `WAIT_PORTS` are coarse feature
  toggles.

Net: "container is the boundary, anything inside goes." Tool calls aren't
gated, prompted, or audited per-action.

## Agent Integration

Agents run *outside*; the sandbox is a remote workspace.

1. Boot container, expose `:8080`.
2. Instantiate `Sandbox(base_url="http://localhost:8080")` (Py/TS/Go SDK).
3. Either call REST methods directly, or have the LLM call `/mcp` tools
   (the README's OpenAI/MiniMax/LangChain/browser-use examples wrap
   `sandbox.shell.exec_command` / `sandbox.jupyter.execute_code` as a
   single host-side tool). For browser automation, agents connect Playwright
   to the in-container Chromium via CDP: `sandbox.browser.get_info().cdp_url`.
4. Optional: the human attaches via `/vnc/`, `/code-server/`, or
   `/v1/docs` to observe.

## Lifecycle

- Boot ready when `WAIT_PORTS=8079,8091` are listening (MCP hub + REST API).
- State lives on the container filesystem at `/home/gem`. Persistence is
  whatever the operator volume-mounts (`docker-compose` example mounts
  `/tmp/gem/vite-project:/home/gem/vite-project`).
- No first-class snapshot/reset API in the SDK. The "reset" pattern is
  `docker rm` + `docker run` (the quick-start uses `--rm -it`). The Volces
  / VeFaaS hosted variant presumably handles snapshotting outside the OSS.

## Failure Modes

- Container OOM (`mem_limit=8g`) or crash → all sessions, browser state,
  shell PIDs, and unmounted workspace files lost. No HA.
- Network drop → REST returns errors; SSE streams break; agent must
  reconnect. Long-running shell jobs survive in-container but the agent
  loses tail.
- Browser CDP connection churn during page reloads — the `browser_*`
  resource split (`browser_page`, `browser_tabs`, `browser_state`,
  `browser_cookies`, `browser_network`, `browser_captcha`) suggests they
  needed per-aspect reconnection logic.
- No process-level circuit breaker for runaway shells beyond the per-call
  `timeout` / `hard_timeout` parameters.

## Notable Design Ideas

- **Single-port fan-out behind a reverse proxy** (`/v1`, `/mcp`, `/vnc`,
  `/code-server`, `/v1/docs`). Cuts firewall/TLS surface to one port —
  appealing for our broker if we ever add a UI.
- **Fern-generated multi-language SDKs from one OpenAPI spec** — Py + TS
  + Go are all auto-generated, low maintenance.
- **MCP exposed as HTTP RPC with `list_mcp_servers` + `list_mcp_tools` +
  `execute_mcp_tool`**. Worth borrowing as a *broker introspection* shape:
  even if our broker isn't an MCP host, having a uniform "list policies /
  invoke policy" REST mirror is useful for hooks.
- **Resource splitting on browser** (page/tabs/cookies/state/network/
  captcha) — fine-grained REST surface where each resource has clear
  lifecycle, vs one mega `browser` blob.
- **`str_replace_editor` parity** with Anthropic's tool spec — the
  sandbox lets the agent's existing tool-calling vocabulary run unchanged.
- **`PROXY_SERVER` + `DNS_OVER_HTTPS_TEMPLATES` env** — egress steering
  via env vars, no code, easy to slot a corporate proxy in front.

## Anti-Patterns / Caveats

- **"One container = boundary" is brittle for our use case.** The threat
  model is "isolate the agent from the host", not "isolate the agent from
  itself." Once a tool call lands inside, every other tool inherits the
  same authority — there is no sandbox-broker-style per-Bash / per-Edit
  decision. Combined with `seccomp=unconfined`, a kernel exploit or any
  in-container privilege confusion compromises everything. Our per-tool
  hook approach (claude-code-hook.sh, codex-hook.sh) is strictly more
  expressive, at the cost of needing host integration.
- **No audit trail in the SDK.** Tool calls go straight through. An
  in-house broker should log decisions; AIO Sandbox doesn't, at this
  layer.
- **Bundling Jupyter + VSCode + VNC + browser into the agent runtime
  conflates "workspace" and "agent infra."** Heavy image (`8g` RAM, 4
  CPU baseline), slow cold-start vs a thin permission broker. Reasonable
  for an "agent VM" product, wrong shape if the goal is a guard between
  the agent and a developer's real machine.
- **No declarative policy file.** Compare to our Nix-driven config —
  AIO is all env vars + runtime `proxy.add_mapping`, harder to review.
