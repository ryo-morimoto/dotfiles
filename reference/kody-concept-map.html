Kody Concept Map

Kodyを動かす前に使う圧縮版の概念地図。Hermes比較、実装判断、`start_coding_agent` の置き場所を決めるための参照用。

One sentence

現時点の証拠では、Kodyはagent本体というより、MCP経由で使える個人用の道具・記憶・secret・package・job実行基盤として捉えるのが安全。

Core split

Kodyが持つ

MCP endpoint、`search`、`execute`、capability registry、saved packages、secrets、values、memory、jobs、workflows、services、repo sessions。

中心には見えない

LLM agent loop、planner/executor/reviewer orchestration、coding agent runtime、モデル選択、agent step履歴。少なくともKodyの主要primitiveとしては前面に出ていない。

外部候補

Cursor Cloud Agents、Codex cloud agent、GitHub Actions、その他の実作業agentは外部に任せる候補。Kodyは文脈と鍵を渡すcontrol planeになれる可能性がある。

Decision rule

Core capability

Kody全体の標準primitiveにしたい、全ユーザーに共通、強い永続化/監査/検索統合が必要。

Saved package

特定workflow、外部API連携、試行錯誤、ユーザー単位のsecret/storageで足りそうな場合。MVP候補。

External agent

repoをcloneして実装する、長い推論loopを回す、PRを作る、CIを直す。Kody外の実行基盤に任せる候補。

`start_coding_agent` placement

最初の実験先は saved package が有力。`cursor_api_key` と GitHub token は saved secret に置き、package-owned storage に `runId`、`agentId`、repo、task digest、statusを保存する。複数workflowで繰り返し使う標準機能になったら core capability化を検討する。
