# Linear Issue Triage

Triage issues in the current Linear workspace using a workflow-status state machine. Use the Linear MCP server for all Linear operations.

Ported from mattpocock's `github-triage` skill. Key differences:

- **State** is carried by Linear's **workflow status** (not a label). Linear already models Triage / Backlog / In Progress / Done / Canceled — use those, and add a handful of triage-specific states as needed.
- **Category** (bug / enhancement) is carried by **labels**.
- Issues are identified by Linear **identifier** (e.g. `ENG-123`), not a number.

## AI Disclaimer

Every comment posted to Linear during triage **must** include the following disclaimer at the top of the comment body, before any other content:

```
> *This was generated by AI during triage.*
```

## Setup: confirm the triage workspace

Before the first triage action in a session:

1. Ask the user (or infer from repo context) which Linear **team** to triage.
2. Call `list_issue_statuses` for that team. Linear teams come with a default set of statuses. A minimal triage setup needs at least:

   | Triage state         | Linear status (suggested mapping)            |
   | -------------------- | -------------------------------------------- |
   | `needs-triage`       | built-in **Triage** status                   |
   | `needs-info`         | a status named **Needs Info** (custom)       |
   | `ready-for-agent`    | a status named **Ready for Agent** (custom)  |
   | `ready-for-human`    | built-in **Backlog** or **Todo**             |
   | `wontfix`            | built-in **Canceled**                        |

   If the team is missing `Needs Info` or `Ready for Agent`, tell the user and offer to proceed with the closest existing status, or pause so they can add the custom ones in Linear's UI. Do not silently remap.

3. Call `list_issue_labels` for that team. Confirm `bug` and `enhancement` (or the team's equivalents) exist. If not, flag it.

Record the resolved status IDs and label IDs in memory for the rest of the session.

## State Machine

Every issue should be in **one** workflow status and have **one** category label (`bug` or `enhancement`).

| Current State     | Can transition to  | Who triggers it        | What happens                                                                                                                                   |
| ----------------- | ------------------ | ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| `no status`       | `needs-triage`     | Skill (on first look)  | Issue needs maintainer evaluation. Skill moves it to Triage after presenting recommendation.                                                   |
| `no status`       | `ready-for-agent`  | Maintainer (via skill) | Issue is already well-specified and agent-suitable. Skill writes agent brief comment, sets status.                                             |
| `no status`       | `ready-for-human`  | Maintainer (via skill) | Issue requires human implementation. Skill writes a brief comment summarizing the task, sets status.                                           |
| `no status`       | `wontfix`          | Maintainer (via skill) | Spam, duplicate, or out of scope. Skill cancels the issue with a comment (and writes `.out-of-scope/` for enhancements).                       |
| `needs-triage`    | `needs-info`       | Maintainer (via skill) | Issue is underspecified. Skill posts triage notes capturing progress so far + questions for reporter, sets status.                             |
| `needs-triage`    | `ready-for-agent`  | Maintainer (via skill) | Grilling session complete, agent-suitable. Skill writes agent brief comment, sets status.                                                      |
| `needs-triage`    | `ready-for-human`  | Maintainer (via skill) | Grilling session complete, needs human. Skill writes a brief comment summarizing the task, sets status.                                        |
| `needs-triage`    | `wontfix`          | Maintainer (via skill) | Maintainer decides not to action. Skill cancels with a comment (and writes `.out-of-scope/` for enhancements).                                 |
| `needs-info`      | `needs-triage`     | Skill (detects reply)  | Reporter has replied. Skill surfaces to maintainer for re-evaluation.                                                                          |

An issue can only move along these transitions. The maintainer can override any state directly (see Quick State Override), but the skill should flag if the transition is unusual.

## Invocation

The maintainer invokes `/linear-triage` then describes what they want in natural language. The skill interprets the request and takes the appropriate action.

Example requests:

- "Show me anything that needs my attention"
- "Let's look at ENG-42"
- "Move ENG-42 to ready-for-agent"
- "What's ready for agents to pick up?"
- "Are there any un-triaged issues?"

## Workflow: Show What Needs Attention

When the maintainer asks for an overview, query Linear with `list_issues` (filter by team, and by each status in turn) and present a summary grouped into three buckets:

1. **No-status issues** — created without a workflow status. These have never been triaged.
2. **`needs-triage` issues** (Triage status) — maintainer needs to evaluate or continue evaluating.
3. **`needs-info` issues with new activity** — the reporter (or anyone other than the triager) has commented since the last triage-notes comment. Use `list_comments` and check timestamps.

Display counts per group. Within each group, show issues oldest first (longest-waiting gets attention first). For each issue, show: identifier, title, age, and a one-line summary of the issue body.

Let the maintainer pick which issue to dive into.

## Workflow: Triage a Specific Issue

### Step 1: Gather context

Before presenting anything to the maintainer:

- Call `get_issue` for the full body, description, labels, assignee, priority, current status.
- Call `list_comments` and read every comment. Parse prior "Triage Notes" comments to see what has already been established in previous sessions.
- Explore the codebase to build context — understand the domain, relevant interfaces, and existing behavior related to the issue.
- Read `.out-of-scope/*.md` files in the current repo (see [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)) and check if this issue matches or is similar to a previously rejected concept.

### Step 2: Present a recommendation

Tell the maintainer:

- **Category recommendation:** bug or enhancement, with reasoning
- **State recommendation:** where this issue should go, with reasoning
- If it matches a prior out-of-scope rejection, surface that: "This is similar to `.out-of-scope/concept-name.md` — we rejected this before because X. Do you still feel the same way?"
- A brief summary of what you found in the codebase that's relevant

Then wait for the maintainer's direction. They may:

- Agree and ask you to apply the state → do it
- Want to flesh it out → start a grilling session
- Override with a different state → apply their choice
- Want to discuss → have a conversation

### Step 3: Bug reproduction (bugs only)

If the issue is categorized as a bug, attempt to reproduce it before starting a grilling session:

- Read the reporter's reproduction steps (if provided)
- Explore the codebase to understand the relevant code paths
- Try to reproduce the bug: run tests, execute commands, or trace the logic
- If reproduction succeeds, report what you found — include the specific behavior you observed and where in the code it originates
- If reproduction fails, report that too — the bug may be environment-specific, already fixed, or the report may be inaccurate
- If the report lacks enough detail to attempt reproduction, note that — this is a strong signal the issue should move to `needs-info`

The reproduction attempt informs the grilling session and the agent brief.

### Step 4: Grilling session (if needed)

If the issue needs to be fleshed out before it's ready for an agent, interview the maintainer to build a complete specification.

### Step 5: Apply the outcome

Depending on the outcome:

- **ready-for-agent** — post an agent brief comment (see [AGENT-BRIEF.md](AGENT-BRIEF.md)) via `save_comment`, then move status with `save_issue`.
- **ready-for-human** — post a comment summarizing the task, what was established during triage, and why it needs human implementation. Use the same structure as an agent brief but note the reason it can't be delegated to an agent (e.g. requires judgment calls, external system access, design decisions, or manual testing). Then move status.
- **needs-info** — post triage notes with progress so far and questions for the reporter (see Needs Info Output below), then move status.
- **wontfix (bug)** — post a polite comment explaining why, then move status to `Canceled`.
- **wontfix (enhancement)** — write to `.out-of-scope/`, post a comment linking to it, then move status to `Canceled` (see [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)).
- **needs-triage** — move to Triage. Optionally post a comment if there's partial progress to capture.

All status transitions are done via `save_issue` with the resolved status ID from the setup step. Always re-read the issue with `get_issue` immediately after to confirm the new status stuck.

## Workflow: Quick State Override

When the maintainer explicitly tells you to move an issue to a specific state (e.g. "move ENG-42 to ready-for-agent"), trust their judgment and apply the state directly.

Still show a confirmation of what you're about to do: which status will be set, which labels will be added/removed, and whether you'll post a comment or cancel the issue. But skip the grilling session entirely.

If moving to `ready-for-agent` without a grilling session, ask the maintainer if they want to write a brief agent brief comment or skip it.

## Needs Info Output

When moving an issue to `needs-info`, post a comment that captures the grilling progress and tells the reporter what's needed:

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

Include everything resolved during the grilling session in "established so far" — this work should not be lost. The questions for the reporter should be specific and actionable, not vague ("please provide more info").

When passing markdown to `save_comment`, send real newlines (not `\n`) per the linear MCP instructions.

## Resuming Previous Sessions

When triaging an issue that already has triage notes from a previous session:

1. Read all comments to find prior triage notes
2. Parse what was already established
3. Check if the reporter has answered any outstanding questions
4. Present the maintainer with an updated picture: "Here's where we left off, and here's what the reporter has said since"
5. Continue the grilling from where it stopped — do not re-ask resolved questions
