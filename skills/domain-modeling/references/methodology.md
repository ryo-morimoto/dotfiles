# Domain Modeling — 方法論リファレンス

## 用語定義

### Domain（問題空間）

ソフトウェアが解こうとする対象領域。具体的には、そのソフトウェアが扱う**データの種類**、**データ間の関係**、**データに対する操作**、**操作に課される制約**の4つの集合。

例：
- コンパイラの問題空間＝ソースコード文字列、トークン列、抽象構文木、型情報、機械語バイナリ、それらの間の変換規則と制約
- ネットワークプロトコルスタックの問題空間＝接続状態、パケット構造、状態遷移規則、タイムアウト制約

「ドメインエキスパート」「ビジネスロジック」等の用語は使用しない。問題空間の知識を持つ人間を「情報源」と呼ぶ。

### Domain Model（型と関数の集合）

問題空間の構造を、プログラミング言語の型定義と関数シグネチャで表現したもの。3種類の要素で構成：

1. **型定義**：問題空間のデータ構造を記述。構成方法は3つのみ。
   - **積型（AND）**：AかつBかつC。レコード、構造体、タプル。
   - **和型（OR）**：AまたはBまたはC。enum、sealed interface、discriminated union。
   - **ラッピング型**：既存の型に名前と生成時制約を付与。`type Email = string`に対して`parseEmail: string → Result<Email, Error>`を通じてのみ生成可能。

2. **関数シグネチャ**：問題空間の操作を記述。`InputType → OutputType`の形式。副作用がある場合は戻り値型に含める（例：`Result<T, E>`、`Option<T>`、`IO<T>`）。

3. **モジュール構造**：型と関数のグルーピング。問題空間の部分領域ごとにモジュールを分ける。

### Domain Modeling（反復的な型の設計行為）

問題空間の構造を型定義と関数シグネチャに変換し、コンパイラで検証可能な形にする行為。一回で完成しない。問題空間の理解は実装を通じて深まるため、型定義の反復的改訂を前提とする。

---

## 前提条件

- 問題空間の定義は**不完全**。分かっている部分と分かっていない部分が混在する。
- 不完全であること自体は障害ではない。不完全さを**可視化**し、反復ごとに**減少を計測**する。
- 静的型検査を持つ言語を使用する（TypeScript、Rust、F#、Scala、Kotlin、Haskell、Swift等）。

---

## 測定指標

各反復ごとに計測し記録する。進捗の客観的判定に使う。

| 指標 | 定義 | 計測方法 | 期待される傾向 |
|---|---|---|---|
| **Placeholder数** | `TODO`/`unknown`/`any`型の出現数 | コード内grep | 単調減少 |
| **外部分岐数** | 型のOR分岐ではなくif/switchで処理している条件分岐の数 | 型の外にあるドメイン概念の分岐をカウント | 単調減少 |
| **シグネチャ嘘数** | 関数の戻り値型が実際の返却値域と一致しない箇所の数 | 戻り値型がstringやanyなのに実際は限定された値域を持つ関数をカウント | 単調減少 |
| **型外ルール数** | コメントやドキュメントにのみ存在し、型に反映されていない制約の数 | コメント中の「〜でなければならない」「〜の場合のみ」をカウント | 単調減少 |
| **コンパイルエラー数** | 型変更後のコンパイルエラー | コンパイラ出力 | 改訂直後に増加→解消 |

## 手順

### Phase 0：問題空間の棚卸し

**入力**：問題空間に関して現時点で得られる情報（仕様書、情報源へのヒアリング、既存コード、プロトコル仕様等）

**作業**：以下を列挙する。

1. このソフトウェアが受け取る**入力データ**の一覧
2. このソフトウェアが生成する**出力データ**の一覧
3. 入力から出力に至る**変換ステップ**の一覧（粒度は粗くてよい）
4. 各データ・各ステップに課される**制約**の一覧

各項目に以下のタグを付与する。

| タグ | 意味 | 基準 |
|---|---|---|
| `KNOWN` | 確定している | 情報源または仕様書で裏付けがある |
| `HYPOTHESIS` | 仮説 | 合理的な推測だが裏付けなし |
| `UNKNOWN` | 不明 | 情報が存在しない |

**出力**：タグ付き項目リスト

**検証**：
- 全項目にタグがある → Yes/No
- `UNKNOWN`項目が0個でないことを確認（0個の場合、見落としの可能性を疑う）

---

### Phase 1：初期型定義

**入力**：Phase 0の項目リスト

**作業**：`KNOWN`と`HYPOTHESIS`の項目を型定義に変換する。`UNKNOWN`の項目は以下のいずれかで処理する。

```typescript
// 方法A：プレースホルダー型
type UnknownPaymentResult = { _placeholder: "UNKNOWN: 決済結果の構造が未定義" }

// 方法B：最も制約の緩い型 + TODOコメント
type PaymentResult = unknown // TODO: 決済API仕様確認後に定義
```

変換規則：

| 問題空間の構造 | 型の構成 | 例 |
|---|---|---|
| AはBとCを持つ | 積型 | `type Order = { id: OrderId, items: OrderItem[] }` |
| AはBまたはCである | 和型 | `type PaymentMethod = CreditCard \| BankTransfer \| Cash` |
| Aは制約付きの基本型 | ラッピング型 + 生成関数 | `type PortNumber = number` + `parsePort: number → Result<PortNumber, Error>` |
| AをBに変換する | 関数シグネチャ | `type Validate = UnvalidatedOrder → Result<ValidatedOrder, ValidationError>` |

**出力**：コンパイル可能な型定義ファイル + プレースホルダー一覧

**検証**：
- コンパイルが通る → Yes/No
- プレースホルダー数を記録 → 数値

---

### Phase 2：最小パス実装

**入力**：Phase 1の型定義

**作業**：入力→出力を1本だけ、実際に動くコードとして実装する。型定義が実際のデータフローと整合するかの最初の検証。

選択基準：最も理解度が高い（`KNOWN`タグが最も多い）パスを選ぶ。

**出力**：1本のend-to-endで動作する実装

**検証**：
- テストが1件以上通る → Yes/No
- 実装中に型の不整合が発生した箇所を記録 → 不整合リスト

---

### Phase 3：不整合の検出と分類

**入力**：Phase 2の実装 + 不整合リスト

**作業**：以下の4種類の不整合をコードから検出する。

| 不整合の種類 | 検出方法 | 例 |
|---|---|---|
| **型不足** | 型に収まらないランタイム値が存在する | `string`型の変数に実際は3種類の固定値しか入らない |
| **型肥大** | 1つの型が複数の文脈で異なる意味を持つ | `User`型が認証・注文・配送で別の属性セットを使う |
| **分岐漏れ** | if/switchが型のOR定義ではなくロジック内に散在する | `if (status === "active")`が型定義と独立して存在 |
| **シグネチャ不一致** | 関数の型が実際の入出力と合っていない | 戻り値`string`だが実際は`null`を返す場合がある |

各不整合に対して以下を記録する：

```
- ファイル: src/order.ts:42
- 種類: 型不足
- 現状: status: string
- 必要なアクション: 和型に変更 → type OrderStatus = "pending" | "confirmed" | "shipped"
```

**出力**：不整合リスト（ファイル位置、種類、必要なアクションの3項目を持つ）

**検証**：
- 全項目にファイル位置がある → Yes/No
- 全項目にアクション種別がある → Yes/No
- 外部分岐数・シグネチャ嘘数・型外ルール数を計測 → 数値

---

### Phase 4：型の改訂

**入力**：Phase 3の不整合リスト

**作業**：不整合リストの各項目に対して型定義を変更する。型変更の種類は以下の4つのみ。

| 操作 | 内容 | コンパイラの反応 |
|---|---|---|
| **分割** | 1つの型を2つ以上の型に分ける | 旧型を使用する全箇所でエラー → 各箇所で新型を選択 |
| **統合** | 2つ以上の型を1つにまとめる | 旧型の使用箇所で型不一致エラー → 統合型に置換 |
| **制約追加** | ラッピング型の生成関数に制約を追加する | 生成関数の呼び出し元でResult処理が必要になる |
| **暗黙概念の型化** | コメントやif文に存在するルールを型として明示する | 新型に移行する箇所でエラー |

手順：
1. 型定義を変更する
2. コンパイルする → エラー一覧を取得
3. 各エラー箇所を修正する
4. 全エラーが解消されるまで2-3を繰り返す
5. テストを実行する

**出力**：更新された型定義 + 通過テスト一覧

**検証**：
- コンパイルが通る → Yes/No
- 既存テストが全通過 → Yes/No
- プレースホルダー数がPhase 1以下 → Yes/No（数値で記録）
- 外部分岐数がPhase 3以下 → Yes/No（数値で記録）

---

### Phase 5：構造的欠陥の判定

**入力**：Phase 3-4を複数回繰り返した結果

**作業**：以下の条件のいずれかが成立するか判定する。

| 条件 | 意味 |
|---|---|
| 3回以上の反復で同じ型が不整合リストに出現する | その型の設計前提が誤っている |
| 不整合リストの複数項目が同一の根本原因を共有する | 型構造自体の再設計が必要 |
| 新しいパスを実装しようとするとプレースホルダーが増加する | 問題空間の理解に構造的な穴がある |
| 情報源が使う概念と型名が対応しない | 問題空間の分割方法を見直す必要がある |

いずれかが成立する場合：Phase 0に戻り、問題空間の棚卸しをやり直す。型構造の大部分を書き直す。

いずれも成立しない場合：Phase 2に戻り、次のパスを実装する。

**出力**：判定結果（Phase 0に戻る / Phase 2に戻る）+ 根拠

---

## フィードバックループの構造

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3 ──→ Phase 4 ──┐
  ↑                                                          │
  │          ┌──────────────────────────────────────────────┘
  │          ↓
  │     Phase 5（判定）
  │          │
  │          ├─→ 構造的欠陥なし → Phase 2（次のパス）
  │          │
  └──────────┴─→ 構造的欠陥あり → Phase 0（再棚卸し）
```

**ループ不変条件**：各反復の終了時に以下が成立すること。

1. コンパイルが通る
2. 既存テストが全通過する
3. プレースホルダー数が記録されている
4. 外部分岐数が記録されている

**終了条件**：以下のすべてが成立した時点でモデリングを完了とする。

1. プレースホルダー数が0
2. 全ユースケースのパスが実装されている
3. 外部分岐数が0（全てのドメイン概念の分岐が型のOR定義で表現されている）
4. シグネチャ嘘数が0

完全な終了は現実にはまれ。プレースホルダー数と外部分岐数の減少が停滞した時点で、残存項目を技術的負債として記録し、次の開発サイクルに持ち越す。

## 原則

型の設計時に適用する原則。原則ごとに**違反の検出方法**を付記する。

### 原則1：不正な値を型で表現不可能にする

string、int、booleanの組み合わせで表現すると、型が許容する値域が実際に有効な値域より広くなる。型の値域を問題空間の有効な値域と一致させる。

**違反の検出方法**：型のインスタンスを任意に生成したとき、問題空間で無効な値が生成可能か。可能なら違反。

```typescript
// 違反：EmailとPhoneが両方nullになれる（少なくとも1つ必要という制約が型にない）
type Contact = { email: Email | null; phone: Phone | null }

// 修正：和型で制約を表現
type ContactInfo =
  | { tag: "email"; email: Email }
  | { tag: "phone"; phone: Phone }
  | { tag: "both"; email: Email; phone: Phone }
```

### 原則2：検証ではなく変換で境界を守る

外部からのデータは`unknown`または`string`として受け取り、型付きの値に**変換**する。booleanを返す検証関数ではなく、`Result<TypedValue, Error>`を返す変換関数を使う。変換に成功した値は、以降のコードで再検証不要。

**違反の検出方法**：同じ制約の検証がコードの複数箇所に存在するか。存在すれば違反。

```typescript
// 違反：検証と使用が分離。検証をスキップして使用できる。
function isValidEmail(s: string): boolean { ... }
function sendEmail(to: string) { ... }

// 修正：変換関数でEmail型を生成。Email型はparseEmailを通じてのみ生成可能。
function parseEmail(s: string): Result<Email, EmailParseError> { ... }
function sendEmail(to: Email) { ... }
```

### 原則3：状態遷移を型で表現する

エンティティの状態遷移を、同一の型にstatusフィールドを持たせるのではなく、状態ごとに異なる型を定義し、遷移を関数シグネチャで表現する。

**違反の検出方法**：status/state/phaseという名前のフィールドが存在し、かつそのフィールドの値によって他のフィールドの有効性が変わるか。変わるなら違反。

```typescript
// 違反
type Order = { status: "pending" | "shipped"; deliveryDate?: Date }

// 修正
type PendingOrder = { items: OrderItem[] }
type ShippedOrder = { items: OrderItem[]; deliveryDate: Date; trackingId: TrackingId }
type Ship = (order: PendingOrder, tracking: TrackingId) => ShippedOrder
```

### 原則4：同じ名前の型は同じ文脈でのみ使う

1つの型が複数の文脈で異なる属性セットを使用している場合、文脈ごとに別の型を定義する。

**違反の検出方法**：ある型のフィールドのうち、特定の文脈でのみ参照されるフィールドが存在するか。存在すれば違反。
